# Parachord Protocol — Play Handlers Design

## Overview

Add four `parachord://` deep-link commands so external sites (primarily ListenBrainz, but anyone) can embed "Play in Parachord" buttons on albums, playlists, radio links, and listen-along share targets.

The four new commands extend the existing protocol surface (chat, import, navigation, single-track play) with playback-oriented actions that resolve to a tracklist via the same shared input shapes.

## URL surface

```
parachord://play/album      ?mbid|spotify|applemusic | artist&title | url | tracks  [&shuffle=1]
parachord://play/playlist    ?url | tracks | (artist&title)                          [&shuffle=1]
parachord://play/radio       ?url | tracks[&refill] | artist[&title]                 [&shuffle=1]
parachord://listen-along     ?service=listenbrainz|lastfm  &user=<username>
```

All four commands accept a flexible identifier set. Resolution rules in priority order:

1. **`mbid`** (album only) — MusicBrainz release-group ID. Fast, exact lookup via existing `fetchReleaseData`.
2. **`spotify` / `applemusic`** (album only) — provider catalog IDs.
3. **`url`** — XSPF or LB-style JSON endpoint. Fetched and parsed; for `play/radio`, this URL is also reused as the refill source unless `refill=` is provided.
4. **`tracks`** — base64-encoded JSON array `[{artist, title, album?, mbid?, isrc?}]`. Same shape as the existing `import?tracks=`.
5. **`artist`+`title`** — fuzzy fallback. For album: search MusicBrainz release-groups (existing `album` handler logic). For radio: seed Parachord's existing in-app spinoff (mode B). Not used for playlist (a playlist needs an explicit list).

After resolution, `play/album` / `play/playlist` set the queue and start playback at index 0. `play/radio` hands its tracks to spinoff as the initial pool with refill semantics (see below).

## Runtime behavior matrix

| Command | Plays through tracklist | Auto-extends pool | Stops on | Confirmation |
|---|---|---|---|---|
| `play/album` | Yes (in order) | No | End of list | None |
| `play/playlist` | Yes (in order, optionally shuffled) | No | End of list | None |
| `play/radio` | Yes (in order, optionally shuffled) | Yes (see refill) | Refill returns empty 3× consecutively, OR initial source has no refill capability and pool exhausts | None |
| `listen-along` | N/A — syncs to remote user | Driven by remote user's now-playing | User stops it, or activates a different track | None |

`save=1` and other library-mutating side effects are intentionally NOT supported. Saving is `parachord://import`'s job; keeping the verbs separate avoids privilege creep on link clicks.

## play/radio refill semantics

Three modes by param shape:

| Param shape | Initial pool | Refill |
|---|---|---|
| `?url=<endpoint>` | Fetched from URL | Re-hits the same URL on top-up (LB lb-radio returns fresh batches per call) |
| `?tracks=<inline>&refill=<endpoint>` | Inline, instant first-play | Refills from `refill` URL |
| `?tracks=<inline>` (no refill) | Inline | None — radio ends when pool exhausts |
| `?artist=X[&title=Y]` (mode B) | Generated by Parachord's existing similar-tracks endpoint | Existing in-app spinoff top-up logic |

Refill trigger: same threshold the existing spinoff already uses (`spinoffTracksRef.current.length < N`). When the threshold trips:

1. Fetch the refill URL.
2. Auto-detect response by `Content-Type`:
   - `application/xml` / `application/xspf+xml` → existing XSPF parser
   - `application/json` → look for `tracks[]` or JSPF-style `playlist.track[]`, map to internal track shape
3. Dedupe against the existing pool by `mbid` → `isrc` → `(artist, title)` lowercase.
4. Append survivors to the pool.
5. If 3 consecutive refills return 0 new tracks → stop refilling, let the pool drain naturally.

Soft rate-limit: minimum 5s between refill fetches. Prevents a misbehaving endpoint from getting hammered through us if the threshold logic fires too aggressively.

## Security model

External-share use case → confirmation gates required for **destructive** actions only. Play actions are read-equivalent to clicking a Spotify/AM share link and don't require confirmation.

The existing `import` handler keeps its confirmation dialog (writes to library). The existing `chat` handler keeps its confirmation dialog (sends to AI). The new four are confirmation-free.

### URL validation (for `url` and `refill` params)

Same SSRF guards the `import` handler already enforces:

- Protocol must be `http:` or `https:`.
- Host must not be loopback (`localhost`, `127.0.0.1`, `0.0.0.0`).
- Host must not be in RFC1918 ranges (`10/8`, `172.16/12`, `192.168/16`).
- Host must not end in `.local`.

Same checks applied to `refill` URL on every fetch (not just initial), since a redirect target could shift.

## Implementation map

All work is in `app.js`. Main process protocol parsing already exists (main.js:1540) and dispatches via `onProtocolUrl`.

### New shared helper: `resolveProtocolPlayInput(params, opts)`

Located near the existing `import` handler. Pure async function — no playback side effects.

```js
async function resolveProtocolPlayInput(params, { allowMbid, allowProviderId, allowArtistTitleAlbum }) {
  // Returns { displayName, tracks: [...], albumArt?: string }
  // Throws on hard failures (invalid URL, parse failure).
  // Returns { tracks: [] } with a displayName when the source is well-formed but empty.
}
```

Reuses:
- `fetchReleaseData` for album-by-MBID and album-by-name
- `handleImportPlaylistFromUrl` parsing internals for XSPF
- New JSON-tracklist parser for LB lb-radio responses
- Existing track-shape constructor (`createTrackFromMeta` or equivalent)

### New cases in the protocol switch (app.js:10468+)

```js
case 'play/album':
case 'play/playlist': {
  const { displayName, tracks, albumArt } = await resolveProtocolPlayInput(params, {
    allowMbid: command === 'play/album',
    allowProviderId: command === 'play/album',
    allowArtistTitleAlbum: command === 'play/album',
  });
  if (!tracks.length) { showToast(`Nothing to play: ${displayName}`); break; }
  const ordered = params.shuffle === '1' ? shuffleArray(tracks) : tracks;
  const context = { type: command, name: displayName, albumArt };
  setCurrentQueue(ordered.slice(1));
  await handlePlayRef.current(ordered[0], displayName, context);
  break;
}

case 'play/radio': {
  if (params.artist && !params.tracks && !params.url) {
    // Mode B — seeded spinoff using Parachord's similar-tracks logic
    activateSpinoffRef.current({
      seed: { artist: params.artist, title: params.title || null }
    });
  } else {
    // Mode C — inline tracks and/or URL-based refill
    const { displayName, tracks } = await resolveProtocolPlayInput(params, {});
    if (!tracks.length && !params.url) {
      showToast(`No tracks for radio: ${displayName}`);
      break;
    }
    activateSpinoffRef.current({
      pool: tracks,
      displayName,
      refillUrl: params.refill || params.url || null,
    });
  }
  break;
}

case 'listen-along': {
  const { service, user } = params;
  if (!service || !user) { showToast('listen-along requires service and user'); break; }
  if (!['listenbrainz', 'lastfm'].includes(service)) {
    showToast(`Unknown listen-along service: ${service}`);
    break;
  }
  const friend = friends.find(f =>
    f.service === service && f.username?.toLowerCase() === user.toLowerCase()
  ) || {
    id: `transient:${service}:${user}`,
    service,
    username: user,
    displayName: user,
    transient: true
  };
  activateListenAlongRef.current(friend);
  break;
}
```

### Spinoff API extension

`activateSpinoff` today only accepts a single seed track. Extend its signature to also accept:

- `{ pool: tracks[], displayName, refillUrl? }` — initial pool comes pre-resolved; refill from URL when low (or never, if `refillUrl` is null).

Existing seed-based mode unchanged. The top-up logic branches on `refillUrl` presence:
- `refillUrl` set → fetch + dedupe + append (5s soft rate-limit, 3-empty-fetches stop condition)
- `refillUrl` null AND pool sourced from `pool` → no top-up; let the pool drain
- Existing seed-mode → existing similar-tracks endpoint

### parachord.com/go redirect

**No changes required.** The redirect is URI-encoding-agnostic; it just hands `parachord://...` to the OS. New commands fit the same pattern as long as URLs are properly URL-encoded.

## Failure UX

All errors surface as toasts, never error dialogs (external links should fail passively):

| Scenario | Toast |
|---|---|
| MBID lookup fails | `Album not found` |
| XSPF/JSON URL fetch fails | `Could not load tracks from <hostname>` |
| Empty tracklist after resolve | `Nothing to play: <displayName>` |
| Listen-along service invalid | `Unknown listen-along service: <service>` |
| Listen-along missing params | `listen-along requires service and user` |
| Refill URL fails 3× in a row | (silent, radio just ends) |
| URL fails SSRF check | `Invalid URL: must be public http/https` |

## Test plan

Smoke tests via `open "parachord://..."` from terminal (macOS):

- `play/album?mbid=<known_mbid>` — plays the album in tracklist order
- `play/album?artist=Radiohead&title=OK%20Computer` — same, via search
- `play/album?spotify=<id>` — same, via Spotify catalog
- `play/album?applemusic=<id>` — same, via AM catalog
- `play/album?...&shuffle=1` — same, but shuffled
- `play/playlist?url=<xspf_url>` — fetches XSPF, plays
- `play/playlist?tracks=<base64>` — plays inline list
- `play/radio?url=<lb_radio_endpoint>` — initial pool from URL, refills from same URL when low
- `play/radio?tracks=<base64>&refill=<endpoint>` — inline first, refills from endpoint
- `play/radio?tracks=<base64>` — static pool, ends when exhausted
- `play/radio?artist=Radiohead` — falls through to existing spinoff seeded path
- `listen-along?service=listenbrainz&user=mr_monkey` — activates listen-along with that user
- SSRF: `play/playlist?url=http://localhost/foo` — toast `Invalid URL: must be public http/https`
- Empty refill: mock endpoint returning `{tracks: []}` 3× → radio ends silently

## Out of scope

- `?save=1` on play/playlist (use `parachord://import` for that)
- Provider-specific radio modes other than LB (anyone with a JSON tracklist endpoint already works)
- Confirmation prompts for play actions (decided against — read-equivalent to share links)
- Browser-extension integration (uses internal IPC, not protocol URLs)
- Cross-platform parity for Android (separate work item; the protocol surface itself is platform-agnostic)
